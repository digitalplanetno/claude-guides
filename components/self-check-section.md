# Self-Check Section (Фильтр реальности)

Вставь эту секцию в конец каждого аудита для фильтрации false positives.

---

## САМОПРОВЕРКА (КРИТИЧЕСКИ ВАЖНО!)

**Перед добавлением каждой проблемы в отчёт, задай себе эти вопросы:**

### Фильтр реальности

| Вопрос | Если "нет" → пересмотри severity |
| -------- | ---------------------------------- |
| Это **эксплуатируемо** в реальных условиях? | Теоретическая уязвимость ≠ реальная угроза |
| Есть **путь атаки** для внешнего злоумышленника? | Internal-only ≠ CRITICAL |
| **Какой ущерб** при успешной атаке? | Утечка публичных данных ≠ утечка паролей |
| Требуется ли **аутентификация** для эксплуатации? | Auth-required снижает severity |
| Это **действительно отсутствует** или я не нашёл? | Проверь дважды перед CRITICAL |

### Типичные ложные срабатывания

**Перепроверь перед включением:**

| Кажется проблемой | Почему может не быть |
| ------------------- | --------------------- |
| "Нет auth на endpoint" | Может быть intentionally public (health, webhooks) |
| "CORS: *" | Если endpoint auth-protected — не критично |
| "Sensitive data в логах" | Если логи только на сервере — medium, не critical |
| "Старая версия пакета" | Если нет CVE для этой версии — не security issue |
| "SQL без prepared statements" | Если параметры не от пользователя — не SQLi |
| "Нет комментариев в коде" | Код может быть self-documenting |
| "Длинный файл" | Если логически связан — это нормально |

### Чеклист перед добавлением

```text
□ Я проверил, что это РЕАЛЬНО эксплуатируемо
□ Я понимаю КОНКРЕТНЫЙ путь атаки
□ Severity соответствует РЕАЛЬНОМУ ущербу, а не теоретическому
□ Это не false positive (проверил контекст использования)
□ Рекомендация по исправлению ПРАКТИЧНА и не ломает функционал
```text

### Пример самопроверки

```text
Нашёл: [описание находки]

Проверяю:
- Эксплуатируемо? → [ДА/НЕТ, почему]
- Путь атаки? → [описание]
- Ущерб? → [описание]
- Auth required? → [ДА/НЕТ]
- Действительно проблема? → [вывод]

Решение: [ВКЛЮЧИТЬ/НЕ ВКЛЮЧАТЬ] как [SEVERITY], [обоснование]
```text
